The SQL Hidden Cost of Undirected Relationships

How a missing database constraint led to NHibernate workarounds
and a performance bottleneck at 1,000,000+ records.
Updated March 2, 2026

Context

In our Risk Management system, the Risk Entity served as a central hub with one-to-many associations to Risks, Investments, and Assets.

To keep the grid responsive, we implemented "Count Columns":

  • Columns displayed the number of associations.
  • Clicking a count opened a dialog with the related entities as links.
Query the risk-to-risk associations efficiently to populate these count columns across datasets exceeding one million records.

The table was physically directed (A -> B), but the association was conceptually undirected (A <-> B).

  • The Constraint Problem: Without enforcing a "canonical direction," the database couldn't distinguish between (Risk A, Risk B) and (Risk B, Risk A).
  • ORM Limitations: We used NHibernate, which does not support OR conditions within JOIN predicates—the exact logic needed to check both columns for a Risk ID.

Technical Approach

1. The "Ugly" Workaround

I pair-programmed with a Senior Backend Engineer to find a way around NHibernate's limitations. We moved the OR query out of the join and into a WITH clause (Common Table Expression) to create a derived set.

2. Performance Degradation

While the CTE worked, the OR conditions destroyed index efficiency in Oracle. At a million records, the query slowed significantly because the optimizer could not use simple equality predicates. 

-- Requires OR / CTE
-- Cannot use indexes effectively
SELECT count(*)
FROM Risk_Associations
WHERE Risk_A_ID = 50
   OR Risk_B_ID = 50;
Slow (Full Table/Index Scan)

In Retrospect

With the benefit of perspective, I realized the issue wasn't just NHibernate, it was the data model. We should have enforced a rule where the lower Risk ID is always stored in the first column and the higher in the second.

The "Why"

  • Without Ordering (The OR Problem): If Risk 50 is associated with Risk 10 and Risk 100, the numbers can be in either column and the database needs to check the entire table for both columns at the same time. This usually triggers a Full Table Scan.
  • With Ordering: If Risk 50 is associated with something lower (like 10), 50 must be in the second column. If it is associated with something higher (like 100), 50 must be in the first column.

The SQL

Instead of one big messy OR query, We could use a UNION ALL. This tells th database to perform two separate, lightning-fast Index Seeks.

-- This query is optimized for B-Tree Indexes
SELECT * FROM (
    -- 1. Look ONLY in the first column for 50.
    -- Because of the rule, this finds all associations
    -- where the other ID is > 50.
    SELECT * FROM Risk_Associations WHERE Risk_A_ID = 50

    UNION ALL

    -- 2. Look ONLY in the second column for 50.
    -- Because of the rule, this finds all associations
    -- where the other ID is < 50.
    SELECT * FROM Risk_Associations WHERE Risk_B_ID = 50
) combined_results;
Fast (Index Range Scan)

The Improvement

By moving the "sorting" logic to the INSERT statement (making sure ID_1 < ID_2), you pay a tiny cost once during save to greatly reduce the cost when querying, turning an O(N) problem into O(logN).

The "Ugly" solution was a necessary patch for the moment, but the true fix was an architectural rule: enforce canonical direction to turn an OR performance drain into a simple, indexable EQUALS.
